# 접근 방식 비교: HAPI vs 직접 구성

## 개요

모바일에서 Claude Code를 원격으로 사용하는 두 가지 접근 방식을 비교한다.

- **1안 (HAPI)**: 오픈소스 프로젝트 [HAPI](https://github.com/tiann/hapi)를 사용
- **2안 (직접 구성)**: Cloudflare Tunnel + tmux + ntfy를 조합

## 비교표

| 항목 | 1안: HAPI | 2안: 직접 구성 |
|------|-----------|---------------|
| **모바일 UI** | 웹 UI (승인 버튼, 출력 렌더링) | 터미널 (SSH) |
| **설정 복잡도** | `npx @twsxtd/hapi` 한 줄 | Cloudflare 계정/도메인 + 터널 + tmux + ntfy 개별 설정 |
| **폰에 설치할 앱** | 없음 (브라우저만) | Termius + Ntfy |
| **푸시 알림** | 내장 | ntfy로 별도 구성 (debounce 포함) |
| **승인 방식** | 원탭 버튼 | 터미널에서 직접 타이핑 |
| **음성 입력** | 지원 | 미지원 |
| **다중 에이전트** | 통합 UI에서 관리 | tmux window로 수동 전환 |
| **데이터 위치** | 맥북 로컬 유지 | 맥북 로컬 유지 |
| **네트워크** | 내장 릴레이 (WireGuard+TLS) 또는 Cloudflare Tunnel | Cloudflare Tunnel |
| **맥북 잠자기 방지** | caffeinate 별도 실행 필요 | tmux + caffeinate 통합 |
| **의존성** | Node.js (npx) | cloudflared, tmux |
| **프로젝트 성숙도** | 초기 (2025~, 2,000+ 스타) | 각 도구가 개별적으로 오래 검증됨 |
| **라이선스** | AGPL-3.0 | 해당 없음 (도구 조합) |
| **커스터마이징** | 프레임워크 범위 내 | 자유로움 |

## 1안: HAPI

### 아키텍처

```
┌─ 맥북 ──────────────────────────────┐
│                                      │
│  caffeinate -s                       │
│  └─ hapi (Claude Code를 감싸서 실행) │
│                                      │
└──────────────────────────────────────┘
        │
        │ WireGuard + TLS (릴레이)
        │ 또는 Cloudflare Tunnel
        ▼
┌─ 스마트폰 ───────────────────────────┐
│                                      │
│  브라우저로 HAPI 웹 UI 접속          │
│  ├─ Claude 출력 확인                 │
│  ├─ 승인 버튼 탭                     │
│  ├─ 텍스트/음성 입력                 │
│  └─ 푸시 알림 수신                   │
└──────────────────────────────────────┘
```

### 장점

- **모바일 최적화 UI**: 폰에서 터미널을 다루지 않아도 됨. 승인은 버튼 탭, 출력은 렌더링된 형태로 확인
- **설정 최소화**: `npx @twsxtd/hapi`로 바로 시작. 별도 도구 조합 불필요
- **올인원**: 네트워크 연결, 푸시 알림, 모바일 UI가 모두 내장
- **데이터 로컬 유지**: Happy와 달리 제3자 서버에 코드가 저장되지 않음. 릴레이는 암호화된 패킷만 전달

### 리스크

- **프로젝트 초기 단계**: 장기 유지보수 불확실
- **AGPL-3.0**: 회사 환경에서 라이선스 정책 확인 필요 (단, 사용만 하는 경우 보통 문제 없음)
- **Node.js 의존**: npx로 실행하므로 Node.js가 설치되어 있어야 함
- **블랙박스 요소**: 내장 릴레이의 동작을 완전히 통제하기 어려움 (Cloudflare Tunnel 연동으로 회피 가능)

### 실행 방법

```bash
# 기본 (내장 릴레이 사용)
caffeinate -s -- npx @twsxtd/hapi hub --relay

# Cloudflare Tunnel 연동 시
caffeinate -s -- npx @twsxtd/hapi
# + 별도로 cloudflared 터널 실행
```

## 2안: 직접 구성

### 아키텍처

```
┌─ 맥북 ──────────────────────────────┐
│                                      │
│  tmux (caffeinate -s로 잠자기 방지)  │
│  ├─ Claude Code 세션들               │
│  └─ Claude 훅 → curl ntfy.sh        │
│                                      │
│  cloudflared (SSH 터널)              │
└──────────────────────────────────────┘
        │                    │
        │ HTTPS              │ Cloudflare Tunnel
        ▼                    ▼
   ntfy.sh 서버         스마트폰 SSH
        │
        │ 푸시 알림
        ▼
┌─ 스마트폰 ───────────────────────────┐
│                                      │
│  Ntfy 앱: 알림 수신                  │
│  Termius: SSH → tmux attach          │
└──────────────────────────────────────┘
```

### 장점

- **안정성**: tmux, SSH, Cloudflare는 수년~수십 년 검증된 도구
- **독립성**: 특정 프로젝트에 의존하지 않아 하나가 죽어도 대체 가능
- **투명성**: 전체 구조를 이해하고 있어 문제 해결이 쉬움
- **커스터마이징**: 알림 조건, 세션 관리 등을 자유롭게 조정 가능

### 리스크

- **모바일 UX**: 폰에서 터미널을 다루는 건 본질적으로 불편
- **설정 복잡도**: 여러 도구를 개별적으로 설정하고 연결해야 함
- **관리 포인트**: cloudflared, tmux, ntfy 훅 등 관리할 요소가 많음

## 권장 전략

**1안(HAPI)을 먼저 시도하고, 문제가 있으면 2안(직접 구성)으로 전환한다.**

이유:
1. HAPI가 모바일 UX에서 압도적으로 유리
2. 2안의 인프라(Cloudflare Tunnel, tmux, caffeinate)는 HAPI와 병행하거나 나중에 전환해도 됨
3. HAPI가 불안정하거나 프로젝트가 중단되면 2안으로 즉시 전환 가능
4. 두 방식 모두 데이터를 맥북 로컬에 유지하므로 보안 수준은 동일

## 참고: Happy와의 차이

[Happy](https://github.com/slopus/happy)도 유사한 도구이지만, 중앙집중식 서버에 암호화된 데이터를 저장하는 방식이다. E2E 암호화(4계층 키 구조, AES-256)를 적용하여 서버가 복호화할 수 없지만, 코드가 제3자 서버를 경유한다는 점이 회사 보안 정책과 충돌할 수 있다. HAPI는 데이터가 로컬에 머무는 자체 호스팅 방식으로, 보안 관점에서 더 적합하다.
